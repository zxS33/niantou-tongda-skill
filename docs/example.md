# 示例：把工程卡点转成通达行动

用户输入：

```text
这个 bug 修了两天，测试还是偶发失败。reviewer 又说我这个方向可能不对，我现在很烦，想直接把模块重写了。
```

可以调用 `/tongda-workflows`，选择“工程调试通达”。新的工作流不会停在分析，会在探针后转入 `engineering-execution` 完成交付。

## 可能输出

```markdown
## 玄览明镜表

要澄清的问题：偶发测试失败是否由当前实现方向导致。

| 层次 | 内容 | 证据/来源 | 下一步 |
|---|---|
| 已确认事实 | 测试偶发失败，reviewer 质疑方向 | CI 日志、review 评论 | 固定失败样本 |
| 稳定信号 | 失败非每次出现 | 本地/CI 不一致 | 对比环境 |
| 当前解释 | 可能是并发或时序问题 | 未验证 | 加日志 |
| 情绪噪声 | 想重写模块证明方向没错 | 自我反应 | 暂不作为依据 |
| 未知缺口 | 失败是否和环境、时序、数据种子相关 | 缺记录 | 加最小探针 |

最小验证动作：固定随机种子并记录关键状态转移。
完成信号：能稳定复现或排除一个变量。
```

接着调用 `pojing-breakthrough`：

```markdown
## 破境攻关表

关口定义：找到偶发失败的触发条件。
过关信号：最小复现或明确证伪当前主要假说。

当前假说：失败由异步状态更新顺序导致。
隔离变量：数据种子、执行顺序、环境版本、状态缓存。

最小探针：
- 动作：在状态更新前后打结构化日志，只跑相关测试 100 次。
- 预期：失败样本出现时状态顺序异常。
- 若成功：局部修复状态同步。
- 若失败：转向环境差异假说。

四选一决策：继续。
下一步：先跑探针，不重写模块。

收念句：
我不把 review 当成否定；我先用探针换回事实，再决定是否改方向。
```

如果探针验证了根因，继续调用 `engineering-execution`：

```markdown
## 工程执行卡

交付物：修复偶发失败并补一条回归验证。
当前证据：失败样本中状态更新顺序和断言读取顺序不一致。
本轮边界：只修复异步状态同步，不重写模块，不改公共 API。
最小负责改动：在状态提交后再触发断言读取路径，并补循环测试。
验证矩阵：
- 必跑：相关单测循环 100 次。
- 可选：完整测试套件。
- 无法运行时替代：记录命令和需要的环境。

现在执行：修改目标文件、添加回归测试、运行验证。
```

完成后，agent 应该交付类似结果：

```markdown
已完成：
- 修复异步状态读取顺序导致的偶发失败。
- 新增回归测试覆盖失败路径。

验证：
- `npm test -- state-sync.test.ts`：通过。
- 相关测试循环 100 次：通过。

残余风险：
- 未跑完整 e2e；若此模块依赖浏览器时序，建议在 CI 再观察一轮。
```

## 示例：韩立式长线稳心

用户输入：

```text
reviewer 说我方案太复杂，我有点不服，想把整个模块重构一遍证明这个方向是对的。
```

可以调用 `/hanli-long-game`。

```markdown
## 长线稳心方案

当前诱惑：用大重构证明方案正确。
如果立刻行动，短期会得到：情绪释放和表达能力的机会。
长期可能损失：扩大 diff、增加 review 成本、引入新 bug、关闭回滚路径。

三查：
- 后路：不通过。全量重构会让回滚和 review 都变重。
- 证据：不通过。当前只有争议，没有证明复杂方案必要的数据。
- 复利：不通过。复杂度可能成为维护债。

韩立式五问：
- 小绿瓶在哪里：现有测试、benchmark、相邻模块模式。
- 灵根是什么：时间有限、review 信任不足、模块已有复杂度。
- 后手是什么：先保留现实现，补测试和设计说明。
- 此步值得吗：全量重构暂时不值得。

策略：蓄势

最小长线动作：
- 做什么：补一条行为保护测试，写一段 tradeoff 说明，再提交一个局部简化 diff。
- 保留什么选择权：保留后续重构空间和当前可回滚路径。
- 避免什么债务：避免为了证明自己引入大面积结构变化。
- 何时复查：reviewer 接受测试和 tradeoff 后，再评估是否需要拆小步重构。
```
