# 心结诊断

心结不是情绪本身，而是“事实缺口 + 价值受损 + 未完成动作 + 反复叙事”绑在一起。先拆结，再行动。

## 核心原则

卡住心神的不是困难，而是困难被解释成了自我否定、关系威胁、前途崩塌或不可挽回。诊断不是安慰，是找准卡点。

## 工程用途

- Code review 后觉得自己被否定。
- 一个 bug 修了很久，开始怀疑能力。
- 产品方案被推翻，舍不得沉没成本。
- 设计稿被批“没感觉”，开始防御性解释。
- 团队冲突后反复脑内辩论。

## 不适用场景

- 事实还没整理：先用 `xuanlan-mirror`。
- 已经需要明确取舍：用 `benxin-decision`。
- 有强烈冲动要立刻反击：用 `hanli-long-game`。

## 心结类型

| 类型 | 内心循环 | 工程表现 | 出路 |
|---|---|---|---|
| 悔 | “早知道我就……” | 反复重写历史 | 提炼原则，补救可补部分 |
| 怨 | “凭什么这样对我” | 卡在评审/协作冲突 | 明确边界与沟通动作 |
| 怕 | “万一全完了” | 放大上线/设计/职业风险 | 做最小验证 |
| 羞 | “我是不是不行” | 把问题等同于人格失败 | 改成行为复盘 |
| 亏 | “我投入这么多不能算了” | 不愿删代码、砍需求、换方向 | 只看未来价值 |
| 欲 | “我要证明自己” | 过度重构、争论、炫技 | 转长线稳心 |
| 义 | “我不能辜负他们” | 接过多需求、无法拒绝 | 区分责任与情绪勒索 |

## 方法流程

### 第一步：命名反复念头

原样写出脑中循环的一句话，不加工成漂亮表达。

### 第二步：定位被触动的价值

判断是能力感、尊严、控制感、安全、关系、成果、承诺、未来预期被触动。

### 第三步：拆出未完成动作

心结通常对应一个没完成的动作：
- 要确认事实。
- 要设边界。
- 要道歉或修复。
- 要放弃沉没成本。
- 要做最小验证。
- 要接受不可逆。

### 第四步：判断出路

从五类出路选一个：
- 明镜：事实不清。
- 决断：价值冲突。
- 边界：责任混乱。
- 行动：过度等待。
- 闭环：已发生，需要收念。

## 常见错误

| 错误 | 表现 | 修正 |
|---|---|---|
| 过早劝放下 | 价值还没被看见 | 先说清受损之处 |
| 把心结心理化 | 忽略真实工程问题 | 同时找事实缺口 |
| 沉迷分析 | 诊断后没有动作 | 交给下游 skill |
| 自我攻击 | “我就是不行” | 改成具体行为和条件 |
