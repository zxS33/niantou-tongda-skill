# 质量门禁参考

用风险等级决定验证强度。低风险不仪式化，高风险不省步骤。

## 风险等级

| 等级 | 判断 | 最低门禁 |
|---|---|---|
| R0 | 文案、注释、非关键文档 | 链接/格式自查，必要时预览 |
| R1 | 局部 UI、普通业务逻辑、小工具脚本 | 目标测试或手动验证，检查相关边界 |
| R2 | 多文件功能、共享组件、数据流、构建配置 | 单元/集成测试，构建或类型检查，回归点 |
| R3 | 认证、权限、支付、迁移、删除、并发、安全 | 设计审查、回滚方案、数据备份或 dry-run，多层测试 |

## 完成定义

一个交付可称为完成，必须同时满足：

- 用户目标已被直接处理。
- 改动范围能解释清楚。
- 至少一个验证信号支持结果。
- 残余风险已明说。
- 没有无关改动混入。

## 验证选择

按最贴近风险的顺序选：

1. 复现命令或目标测试。
2. 静态检查：lint、typecheck、schema、markdown link。
3. 构建或集成测试。
4. 浏览器/截图/视觉检查。
5. 人工核对清单。

不要为了显得努力跑无关大测试；也不要因为大测试慢就什么都不跑。

## 不能验证时

必须交代：

- 缺少什么：依赖、凭据、外部服务、设备、时间。
- 已做替代检查：阅读、局部运行、静态推理、样例数据。
- 风险等级：低/中/高。
- 用户下一步：需要谁在什么环境运行什么命令。

## 常见失败模式

| 失败模式 | 迹象 | 修正 |
|---|---|---|
| 表格替代交付 | 输出很清楚但没有产物 | 回到交付物定义并执行 |
| 过度重构 | diff 远大于问题 | 缩回最小负责改动 |
| 验证错位 | 跑了测试但不覆盖改动 | 选择更贴近风险的检查 |
| 忽略宿主约束 | 破坏本地风格或用户改动 | 重新读相邻代码和 git 状态 |
| 办公口径漂移 | 结论漂亮但数字没来源 | 建来源表并标未知 |
