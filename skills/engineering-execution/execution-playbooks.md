# 执行场景速查

根据用户任务选择最小可用流程。不要机械套全套；选能提高交付质量的部分。

## 功能开发

1. 找相似功能、入口、状态管理、测试位置。
2. 写一条用户可见验收标准。
3. 先实现最小路径，再补边界状态。
4. 添加或更新目标测试。
5. 运行最贴近改动的检查。
6. 交付时说明新增能力、覆盖状态和未覆盖范围。

风险信号：
- 需要改公共 API。
- 涉及权限、支付、数据删除、迁移。
- UI 改动可能影响移动端或可访问性。

## Bug 修复

1. 写清复现条件和失败信号。
2. 找最近相关改动、相似 issue、测试缺口。
3. 用最小探针验证一个假说。
4. 修复根因，避免只压错误信息。
5. 加回归测试或复现脚本。
6. 说明复现前后行为差异。

不要因为烦躁而重写模块；先证明重写是必要的。

## Review 跟进

1. 把评论分成必须改、可讨论、误解、范围外。
2. 对必须改项逐条处理。
3. 对不同意项用事实和权衡回复，不用情绪措辞。
4. 每次提交保持主题聚焦。
5. 最后列出已处理评论和仍需 reviewer 决策的点。

## 重构

1. 说明重构目标：可读性、复用、性能、解耦或迁移准备。
2. 先确认现有行为的保护测试。
3. 小步移动，避免功能变化混入结构变化。
4. 保留回滚路径。
5. 用 diff 证明公共行为不变。

重构不是证明能力；重构必须减少未来维护成本。

## 文档与办公交付

1. 明确受众：用户、同事、老板、客户、评审人。
2. 明确用途：决策、执行、培训、归档、说服、复盘。
3. 先列信息来源和不确定口径。
4. 结构先服务阅读路径，不服务作者表达欲。
5. 数字、日期、名字、链接和结论必须复核。
6. 交付时说明哪些内容可直接使用，哪些需要人确认。

## 设计与前端

1. 先确认用户动作和核心状态，不先堆视觉装饰。
2. 找项目现有组件、间距、色彩、图标和响应式规则。
3. 覆盖空、加载、错误、长文本、移动端。
4. 用截图或浏览器验证实际渲染。
5. 交付时说明视觉变化和交互路径。

## 发布与交接

1. 工作树必须清楚：哪些是本轮改动，哪些不是。
2. 提交信息写交付事实，不写情绪。
3. 推送前跑可承担的检查。
4. 交接说明包括变更、验证、风险、回滚或后续。
