# 工程执行契约

本 skill 把“念头通达”落到真实交付。通达不是说清楚就结束；对工程和办公任务，通达必须体现为可检查的产物、验证信号和交付说明。

## 核心原则

心要通，手也要稳。工程执行 = 仓库侦察 + 范围保护 + 最小负责改动 + 验证证据 + 风险交代 + 收念转场。

使用本 skill 时，不要只输出方法论表格。除非用户只要求分析，否则必须推进到文件、代码、命令、文档、设计或明确的下一步交付。

## 不适用场景

- 用户只问概念解释、文案、哲学阐释：用相关思想 skill，不启动执行契约。
- 事实完全混乱且无法定位上下文：先用 `xuanlan-mirror`。
- 任务涉及高风险法律、医疗、财务、安全事故：先走专业安全流程。
- 用户明确说“先别改代码”：只给计划和风险，不动文件。

## 执行流程

### 1. 定义交付物

把用户需求改写成可交付结果：
- 代码：修复什么行为、增加什么能力、保留什么边界。
- 文档：给谁读、解决什么问题、需要什么格式。
- 设计：服务什么用户动作、什么状态必须覆盖。
- 办公：谁是受众、要做什么决策、交付后如何使用。

### 2. 侦察上下文

先读现有系统，不凭空发明风格：
- 用 `rg` / `rg --files` 找入口、测试、配置和相似实现。
- 读相邻代码、README、package scripts、测试命名和本地约定。
- 识别用户已有改动；不要覆盖不属于本轮的变更。
- 对办公产物，先确认模板、受众、口径、数据来源和交付格式。

### 3. 建立证据账本

只把可观察信息当事实：
- 代码事实：文件、函数、测试、错误日志、复现步骤。
- 产品事实：用户反馈、指标、约束、发布日期。
- 设计事实：截图、组件状态、可访问性、视觉层级。
- 办公事实：数据表、会议结论、对方原话、截止时间。

假说必须配验证动作；情绪和偏好不能伪装成技术结论。

### 4. 选择最小负责改动

优先选择能解决问题、便于验证、可回滚的改动：
- 不做无关重构。
- 不扩大公共 API，除非收益和兼容性明确。
- 不为了证明能力做复杂方案。
- 不因短期舒服制造长期维护债。
- 需要重构时，先说明为什么小修不够。

### 5. 执行并保持上下文干净

写代码或产物时：
- 遵循项目已有模式和命名。
- 保持 diff 聚焦。
- 为复杂逻辑加少量有用注释。
- 对用户可见文案、设计和办公材料保持专业口吻。
- 遇到新事实时更新计划，不保护旧假说。

### 6. 验证矩阵

至少选择一个能证明交付有效的验证信号：
- 静态：lint、typecheck、schema validate、链接检查。
- 单元：目标测试、边界测试、回归测试。
- 集成：端到端流程、构建、迁移 dry-run。
- 视觉：截图、响应式检查、可访问性状态。
- 办公：数字口径复核、来源核对、格式预览、受众阅读路径。

如果不能运行验证，必须说明原因、风险和替代检查。

### 7. 交付与收念

结束时只保留用户需要继续行动的信息：
- 改了什么。
- 证明它有效的证据。
- 还剩什么风险。
- 下一步是否需要用户确认。

不要把过程日志塞满最终回答；把关键命令和结果讲清楚即可。

## 操作规程

当任务复杂、跨文件、跨角色或容易空谈时，先输出简短“工程执行卡”，然后立即执行。

```markdown
## 工程执行卡

交付物：……
当前证据：……
本轮边界：……
最小负责改动：……
验证矩阵：
- 必跑：……
- 可选：……
- 无法运行时替代：……

现在执行：……
```

简单任务不要输出卡片，直接做。

## 质量红线

- 不在未读代码时臆造实现。
- 不在未验证时宣布完成。
- 不把大改当果断。
- 不把没跑测试包装成“应该可以”。
- 不覆盖用户或他人未授权改动。
- 不让安慰、热血或哲学语言替代证据。
